## Section A — `Object.create` and prototype links (from Q4)
 
### Q1. What does the following code output?
```javascript
const proto = { greet() { return 'hi'; } };
const a = Object.create(proto);
const b = Object.create(null);
 
console.log(a.greet());
console.log(Object.getPrototypeOf(a) === proto);
console.log(Object.getPrototypeOf(b));
```
- A. `'hi' true null`
- B. `'hi' true undefined`
- C. `TypeError true null`
- D. `'hi' false {}`
> [!success]- Explanation
> - `Object.create(proto)` builds a new object whose `[[Prototype]]` slot points at `proto`. So `a.greet()` resolves through the prototype chain.
> - `Object.getPrototypeOf(a) === proto` is `true` — that's literally what `Object.create` set up.
> - `Object.create(null)` creates a prototype-less object (a "dictionary object" with no `Object.prototype` methods like `toString` or `hasOwnProperty`). Its prototype is **`null`**, not `undefined`.
>
> Use case for `Object.create(null)`: safe maps where you don't want inherited keys to collide with user keys.
>
> ---
>
> **→ Answer: A**
 
---
 
### Q2. Which of the following is **true** about objects created with `Object.create(Constructor.prototype)`?
- A. The object is fully initialized as if `new Constructor()` had been called
- B. The object's `[[Prototype]]` is `Constructor.prototype`, but the constructor body is not run
- C. The object's `[[Prototype]]` is `Constructor` itself
- D. The object cannot use methods defined on `Constructor.prototype`
> [!success]- Explanation
> `Object.create(Constructor.prototype)` only sets up the prototype link. It does **not** invoke the constructor body, so any `this.x = ...` initialization performed inside `Constructor` is skipped.
>
> ```javascript
> function Animal(name) { this.name = name; } // not run!
> const a = Object.create(Animal.prototype);
> console.log(a.name); // undefined
> ```
>
> This is exactly how `new` works internally before running the constructor body — `Object.create(Constructor.prototype)` then `Constructor.call(newObj, ...args)`.
>
> ---
>
> **→ Answer: B**

## Section B — `Object.defineProperty` defaults (from Q5)
 
### Q3. What does the following code output?
```javascript
'use strict';
const obj = {};
Object.defineProperty(obj, 'x', { value: 1 });
try {
  obj.x = 99;
  console.log('assigned:', obj.x);
} catch (e) {
  console.log(e.constructor.name);
}
```
- A. `assigned: 99`
- B. `assigned: 1`

>
> Quick mental rule:
> | Mode       | Assign to non-writable property |
> |------------|--------------------------------|
> | Non-strict | Silent failure (no error)       |
> | Strict     | `TypeError`                    |
>
> ---
>
> **→ Answer: C**

### Q4. After running the snippet, what does `Object.keys(obj)` return?
```javascript
const obj = {};
Object.defineProperty(obj, 'a', { value: 1, enumerable: true });
Object.defineProperty(obj, 'b', { value: 2 });
obj.c = 3;
```
- A. `['a', 'b', 'c']`
- B. `['a', 'c']`
- C. `['c']`
- D. `[]`
> [!success]- Explanation
> `Object.keys` returns **own enumerable string-keyed** properties.
> - `a`: explicitly `enumerable: true` → included.
> - `b`: defaults to `enumerable: false` → **excluded**.
> - `c`: assigned via `obj.c = 3`, which uses the data property defaults of `[[Set]]`: `writable: true`, `enumerable: true`, `configurable: true` → included.
>
> This is the trap: when you go through `defineProperty`, defaults are restrictive (`false`); when you assign directly, defaults are permissive (`true`). Vue 2's reactivity system used `defineProperty` precisely because it needs to intercept those defaults.
>

> **→ Answer: B**

### Q5. Which property descriptor pair is **invalid** and will throw `TypeError`?
- A. `{ value: 1, writable: true }`
- B. `{ get() { return 1; } }`
- C. `{ value: 1, get() { return 1; } }`
- D. `{ get() { return 1; }, set(v) {} }`
> [!success]- Explanation
> A property descriptor is **either** a data descriptor (`value` / `writable`) **or** an accessor descriptor (`get` / `set`) — never both. Mixing `value` with `get` throws:
> ```
> TypeError: Invalid property descriptor. Cannot both specify accessors and a value or writable attribute
> ```
>
> - A: pure data descriptor — fine.
> - B: accessor (getter only) — fine; setter just defaults to `undefined`.
> - D: accessor with both getter and setter — fine.
> - C: mixes `value` with `get` — invalid.
>
> ---
>
> **→ Answer: C**

## Section C — Default parameters & arithmetic coercion (from Q7)
 
### Q6. What does the following code output?
```javascript
function f(a = 1, b = a + 1, c = b + 1) {
  return [a, b, c];
}
console.log(f());
console.log(f(10));
console.log(f(undefined, 20));
```
- A. `[1,2,3] [10,11,12] [1,20,21]`
- B. `[1,2,3] [10,2,3] [undefined,20,21]`
- C. `[1,2,3] [10,11,12] [undefined,20,21]`
- D. Throws `ReferenceError` (default values cannot reference each other)
> [!success]- Explanation
> Default values are evaluated **left to right at call time**, and earlier parameters are visible to later defaults.
> - `f()`: `a = 1`, `b = 1 + 1 = 2`, `c = 2 + 1 = 3`.
> - `f(10)`: `a = 10`, `b = 10 + 1 = 11`, `c = 11 + 1 = 12`.
> - `f(undefined, 20)`: `a` triggers default `1`; `b = 20` (explicit overrides default); `c = 20 + 1 = 21`.
>
> Note: passing `undefined` triggers the default, so the third call gives `[1, 20, 21]`, not `[undefined, 20, 21]`.
>
> ---
>
> **→ Answer: A**

### Q7. What does the following code output?
```javascript
console.log(1 + undefined);
console.log(1 + null);
console.log('1' + null);
console.log('1' + undefined);
```
- A. `NaN 1 '1null' '1undefined'`
- B. `1 1 '1null' '1undefined'`
- C. `NaN 1 '1' '1'`
- D. `undefined null '1null' '1undefined'`
> [!success]- Explanation
> The `+` operator's coercion rules:
> - **Numeric `+`** (both sides coerce to number): `Number(undefined) === NaN`, `Number(null) === 0`. So `1 + undefined === NaN` and `1 + null === 1`.
> - **String `+`** (one side is string): the other side is coerced via `String()`. `String(null) === 'null'`, `String(undefined) === 'undefined'`. So `'1' + null === '1null'` and `'1' + undefined === '1undefined'`.
>
> Direct mental table to memorize:
> | Value       | `Number(v)` | `String(v)`     |
> |-------------|-------------|-----------------|
> | `null`      | `0`         | `'null'`        |
> | `undefined` | `NaN`       | `'undefined'`   |
> | `''`        | `0`         | `''`            |
> | `[]`        | `0`         | `''`            |
> | `[1]`       | `1`         | `'1'`           |
> | `{}`        | `NaN`       | `'[object Object]'` |
>
> ---
>
> **→ Answer: A**

### Q8. What does the following code output?
```javascript
function greet(name = 'world', greeting) {
  return `${greeting}, ${name}!`;
}
console.log(greet('Leo'));
console.log(greet(undefined, 'Hi'));
```
- A. `undefined, Leo! Hi, world!`
- B. `Hi, Leo! Hi, world!`
- C. `undefined, Leo! Hi, undefined!`
- D. `, Leo! Hi, !`
> [!success]- Explanation
> Default parameters apply per-argument, only when that argument is strictly `undefined`. Trailing arguments are independent.
> - `greet('Leo')`: `name = 'Leo'`; `greeting` is **not passed**, so it's `undefined`. Result: `'undefined, Leo!'`.
> - `greet(undefined, 'Hi')`: `name` is `undefined` → default `'world'`; `greeting = 'Hi'`. Result: `'Hi, world!'`.
>
> Lesson: parameters with defaults don't have to come last syntactically — but if you put them in the middle, callers must explicitly pass `undefined` to skip them, which is awkward. Convention: defaulted params last.
>
> ---
>
> **→ Answer: A**

## Section D — `switch` strict equality (from Q11)
 
### Q9. What does the following code output?
```javascript
function check(v) {
  switch (v) {
    case 0:    return 'zero';
    case false: return 'false';
    case '':   return 'empty';
    default:   return 'other';
  }
}
console.log(check(0));
console.log(check(false));
console.log(check(''));
```
- A. `'zero' 'zero' 'zero'` (all loosely equal to 0)
- B. `'zero' 'false' 'empty'`
- C. `'zero' 'false' 'zero'`
- D. `'other' 'other' 'other'`
> [!success]- Explanation
> `switch` uses `===`, **not** `==`. So `0`, `false`, and `''` are all distinct cases even though they're mutually `==`-equal. Each input falls into exactly the matching case.
>
> This is one of the few places in JS where you reliably get strict comparison without writing `===` yourself — useful when matching on values that have lots of `==` collisions.
>
> ---
>
> **→ Answer: B**

### Q10. What does the following code output?
```javascript
const x = 5;
switch (true) {
  case x < 0:    console.log('negative'); break;
  case x === 0:  console.log('zero'); break;
  case x > 0:    console.log('positive'); break;
  default:       console.log('unknown');
}
```
- A. `negative`
- B. `zero`
- C. `positive`
- D. `unknown`
> [!success]- Explanation
> The `switch (true) { case <boolean expression>: ... }` pattern is a JS idiom for chained `if/else if` with shared structure. Each `case` is evaluated and compared against `true` with `===`. Since `x === 5`, only `x > 0` evaluates to `true`, matching the third case.
>
> This pattern works precisely because `switch` uses strict equality and case expressions are evaluated top to bottom.
>
> ---
>
> **→ Answer: C**

## Section E — `{}` parsing in expression vs statement context (from Q13)
 
### Q11. Run as a top-level statement in a Node REPL or browser console (no surrounding parens), what does the following print?
```javascript
{} + []
```
- A. `'[object Object]'`
- B. `0`
- C. `''`
- D. `NaN`
> [!success]- Explanation
> At statement level, the leading `{}` is parsed as an **empty block statement**, not an object literal. The remainder, `+ []`, is then a separate expression statement: `+[]` coerces the array to a number via `''` → `0`. The REPL prints `0`.
>
> Compare:
> ```javascript
> ({} + [])      // '[object Object]'  (parens force expression context)
> let r = {} + []; // '[object Object]'  (RHS is expression context)
> ```
>
> Mnemonic: the parser decides whether `{` opens a block or an object literal based on **where in the grammar** it appears, not on what's inside. At statement start → block.
>
> ---
>
> **→ Answer: B**

## Section F — `Object.is` vs `===` (from Q14)
 
### Q13. Which of the following is **true**?
- A. `Object.is(NaN, NaN) === true` and `NaN === NaN` is `true`
- B. `Object.is(NaN, NaN) === true` and `NaN === NaN` is `false`
- C. `Object.is(NaN, NaN) === false` and `NaN === NaN` is `true`
- D. `Object.is(NaN, NaN) === false` and `NaN === NaN` is `false`
> [!success]- Explanation
> Two key differences between `Object.is` and `===`:
>
> | Comparison         | `===`   | `Object.is` |
> |--------------------|---------|-------------|
> | `NaN` vs `NaN`     | `false` | `true`      |
> | `+0` vs `-0`       | `true`  | `false`     |
> | Everything else    | Same    | Same        |
>
> `Object.is` implements the **SameValue** algorithm (used in spec for things that should treat NaN as equal to itself, e.g., `Array.prototype.includes`). `===` implements **Strict Equality** which has the legacy NaN/zero quirks.
>
> Note: `Map`/`Set` use **SameValueZero**, which treats `NaN === NaN` as equal but treats `+0` and `-0` as equal — a third variant.
>
> ---
>
> **→ Answer: B**

### Q14. What does the following code output?
```javascript
console.log(+0 === -0);
console.log(Object.is(+0, -0));
console.log([NaN].includes(NaN));
console.log([NaN].indexOf(NaN));
```
- A. `true false true 0`
- B. `false true true 0`
- C. `true false true -1`
- D. `true false false 0`
> [!success]- Explanation
> Three different equality algorithms in play:
>
> 1. `+0 === -0` → `true` (Strict Equality treats them as equal).
> 2. `Object.is(+0, -0)` → `false` (SameValue distinguishes them).
> 3. `[NaN].includes(NaN)` → `true` (`includes` uses SameValueZero, treats NaN as equal to NaN).
> 4. `[NaN].indexOf(NaN)` → `-1` (`indexOf` uses Strict Equality, so it can never find NaN).
>
> The `indexOf` vs `includes` discrepancy is exactly why `includes` was added in ES2016.
>
> ---
>
> **→ Answer: C**

### Q15. When does it actually **matter** that `Object.is(+0, -0) === false`?
- A. When summing arrays of numbers
- B. When using `+0` and `-0` as keys in a `Map`
- C. When dividing: `1 / +0 === Infinity`, `1 / -0 === -Infinity`
- D. Both B and C describe situations where the distinction matters
> [!success]- Explanation
> - C is correct on its own: division by `+0` vs `-0` produces different infinities, so the sign of zero is semantically meaningful in arithmetic.
> - B is **incorrect**: `Map` uses SameValueZero, which treats `+0` and `-0` as the **same** key. So `m.set(+0, 'a'); m.set(-0, 'b')` overwrites the same slot.
>
> So D ("both") is wrong because B is wrong. C alone is the answer.
>
> Quick reference of where each algorithm shows up:
> - **Strict (`===`)**: equality operators, `switch`, `indexOf`.
> - **SameValueZero**: `Map`/`Set` keys, `Array.prototype.includes`.
> - **SameValue (`Object.is`)**: explicit `Object.is` calls, internal property descriptor comparisons.
>
> ---
>
> **→ Answer: C**

## Section G — Object immutability levels (from Q20)
 
### Q16. Which row in the table is **incorrect**?
| Method                         | Add new properties | Modify existing values | Reconfigure descriptors |
|--------------------------------|--------------------|------------------------|------------------------|
| A. `Object.preventExtensions`  | ✗                 | ✓                      | ✓                      |
| B. `Object.seal`               | ✗                 | ✓                      | ✗                      |
| C. `Object.freeze`             | ✗                 | ✗                      | ✗                      |
| D. (none — all rows are correct) |                  |                        |                        |
 
> [!success]- Explanation
> All three rows are correct. The progression of restrictions:
> - `preventExtensions`: can't add new properties; existing ones are unchanged.
> - `seal`: `preventExtensions` + makes all properties `configurable: false` (so you can't delete them or reconfigure descriptors), but values stay writable.
> - `freeze`: `seal` + makes all properties `writable: false` (the strongest level).
>
> Detection helpers: `Object.isExtensible`, `Object.isSealed`, `Object.isFrozen`.
>
> ---
>
> **→ Answer: D**

### Q17. What does the following code output?
```javascript
const obj = Object.freeze({
  inner: { count: 0 }
});
obj.inner.count = 99;
obj.inner = { count: -1 };
console.log(obj.inner.count);
```
(Non-strict mode.)
- A. `0`
- B. `99`
- C. `-1`
- D. Throws `TypeError`
> [!success]- Explanation
> `Object.freeze` is **shallow**. The top-level property `inner` is non-writable (so reassigning it silently fails in non-strict mode), but the object that `inner` points to is **not** frozen. So `obj.inner.count = 99` succeeds.
>
> For deep freeze, you need to recurse:
> ```javascript
> function deepFreeze(o) {
>   for (const v of Object.values(o)) {
>     if (v && typeof v === 'object') deepFreeze(v);
>   }
>   return Object.freeze(o);
> }
> ```
>
> ---
>
> **→ Answer: B**

## Section H — `class` and constructor function interop (from Q23)
 
### Q18. Which of the following is **valid**?
```javascript
function OldStyle(name) { this.name = name; }
OldStyle.prototype.greet = function() { return `Hi, ${this.name}`; };
 
class NewStyle extends OldStyle {
  constructor(name, age) {
    super(name);
    this.age = age;
  }
}
```
- A. Invalid: `class` cannot `extends` a non-class function
- B. Valid: a class can extend any constructible function
- C. Valid only if `OldStyle` has a `prototype.constructor` set explicitly
- D. Valid syntactically but `super(name)` will throw at runtime
> [!success]- Explanation
> A class can `extends` **any constructible value** — that includes legacy constructor functions. The reverse is **not** true: a constructor function cannot cleanly inherit from a class without going through prototype manipulation, because `class` requires `super()` to be called before accessing `this`, and that machinery isn't available to plain function-based "subclasses".
>
> Practical use: this is how you incrementally migrate legacy codebases to `class` syntax — wrap the old hierarchies one layer at a time.
>
> ---
>
> **→ Answer: B**

### Q19. Which statement about `class` semantics that **differ** from a plain constructor function is **incorrect**?
- A. The class body is implicitly in strict mode
- B. Calling a class without `new` throws `TypeError`
- C. Class declarations are hoisted in the same way as `function` declarations
- D. Methods defined in the class body are non-enumerable
> [!success]- Explanation
> - A: correct. Class body is strict by default.

## Section I — `WeakSet` / `WeakMap` constraints (from Q27)
 
### Q20. Which operation **does not** exist on `WeakMap`?
- A. `set(key, value)`
- B. `get(key)`
- C. `has(key)`
- D. `keys()`
> [!success]- Explanation
> `WeakMap` has only: `set`, `get`, `has`, `delete`. **No** `keys`, `values`, `entries`, `forEach`, `size`, or `clear`. The omission is deliberate — exposing iteration would let you observe garbage collection, which is non-deterministic.
>
> Same restriction for `WeakSet`: only `add`, `has`, `delete`. No `size`, no iteration, no `clear`.
>
> ---
>
> **→ Answer: D**
 
---
 
### Q21. Which is a **valid** `WeakMap` key?
- A. `'foo'` (a string)
- B. `42` (a number)
- C. `Symbol('s')` (a symbol — registered Symbols excluded)
- D. `{ id: 1 }` (an object)
> [!success]- Explanation
> In ES2019, `WeakMap` keys must be **objects** (or non-registered Symbols, but that's an ES2023 addition — outside our scope here). Primitives like strings, numbers, and booleans cannot be weakly held because they don't have unique identity to be GC'd against.
>
> ```javascript
> new WeakMap().set('foo', 1); // TypeError: Invalid value used as weak map key
> ```
>
> The classic use case: associate metadata with DOM nodes or arbitrary objects without preventing them from being GC'd when the rest of your code drops its references.
>
> ---
>
> **→ Answer: D**

### Q22. Why does `WeakSet` have **no** `forEach` method?
- A. It was simply forgotten in the spec
- B. Iteration order would expose GC timing, which the spec wants to keep non-observable
- C. `forEach` is incompatible with weak references at the engine level
- D. It does exist, just under a different name
> [!success]- Explanation
> The omission is intentional. If `WeakSet` exposed iteration (or `size`, or `clear`), the order and presence of elements would let observers detect garbage collection. That would (1) make GC behavior part of the observable program semantics — a big constraint on engine implementers — and (2) leak information that could be used as a side channel.
>
> `WeakSet` is meant for "is this object tagged?" membership tests, not for enumeration.
>
> ---
>
> **→ Answer: B**

## Section J — Object literals are not scopes; arrow `this` (from Q28)
 
### Q23. What does the following code output?
```javascript
const greeting = 'Hello';
const obj = {
  greeting: 'Hi',
  arrow: () => this,
  method() { return this; },
};
 
console.log(obj.arrow() === obj);
console.log(obj.method() === obj);
```
(Run at module top level in strict mode, where top-level `this` is `undefined`.)
- A. `true true`
- B. `false true`
- C. `true false`
- D. `false false`
> [!success]- Explanation
> Object literals are **not scopes**. `arrow` is defined in the surrounding lexical scope (the module top level), so its `this` is whatever the enclosing scope's `this` is — `undefined` in strict-mode modules. So `obj.arrow() === obj` is `false`.
>
> `method` is a regular method; `this` is set by the call site `obj.method()`, so it equals `obj`.
>
> Mnemonic: arrow functions don't care **how** they're called — only **where** they were defined.
>
> ---
>
> **→ Answer: B**

### Q24. What does the following code output?
```javascript
function outer() {
  const arrow = () => this;
  return arrow;
}
const fn = outer.call({ tag: 'A' });
console.log(fn().tag);
console.log(fn.call({ tag: 'B' }).tag);
```
- A. `'A' 'B'`
- B. `'A' 'A'`
- C. `'B' 'B'`
- D. `undefined undefined`
> [!success]- Explanation
> The arrow's `this` is captured from `outer`'s execution context at the moment `outer` runs. `outer.call({ tag: 'A' })` sets `this` to `{ tag: 'A' }` inside `outer`, so the arrow closes over that.
>
> Crucially: **`call`/`apply`/`bind` cannot reassign an arrow function's `this`**. The second `fn.call({ tag: 'B' })` ignores the `thisArg` and the arrow still returns `{ tag: 'A' }`.
>
> This is why arrow functions are unsuitable as methods you want to rebind — and why they're great as callbacks where you want to lock in the surrounding `this`.
>
> ---
>
> **→ Answer: B**

### Q25. Which of the following correctly produces `42` when executed?
- A. `const o = { x: 42, get: () => this.x }; o.get();`
- B. `const o = { x: 42, get() { return this.x; } }; o.get();`
- C. `const o = { x: 42, get: function() { return this.x; } }; o.get.call(null);` (strict mode)
- D. `const o = { x: 42 }; (() => o.x).call({ x: 99 });`
> [!success]- Explanation
> - A: arrow function's `this` is the enclosing scope's `this`, not `o`. Returns `undefined` (or `window.x` in non-strict browser, but typically `undefined`).
> - B: shorthand method — when called as `o.get()`, `this === o`. Returns `42`. ✓
> - C: in strict mode, `call(null)` sets `this` to `null`. Then `null.x` throws `TypeError`.
> - D: this is convoluted but correct in producing `42` — the arrow body just reads `o.x`, not `this.x`. But this works for a different reason than "this binding". The question asks which one *correctly produces 42*; both B and D evaluate to 42, but D doesn't actually demonstrate `this` binding (it sidesteps it). B is the canonical correct answer that uses normal `this`.
>
> > Reasoning: B is the option that demonstrates the standard, idiomatic pattern. D, strictly speaking, also evaluates to `42` but via lexical capture of `o`, not via `this`. If a question lists multiple technically-passing options, B is the one matching the test's intent.
>
> ---
>

## Section K — Constructor + arrow callback for stable `this` (from Q29)
 
### Q26. After 1 second, what is `obj.value`?
```javascript
function Box() {
  this.value = 0;
  setTimeout(function() { this.value = 99; }, 1000);
}
const obj = new Box();
```
(Browser, non-strict mode.)
- A. `99`
- B. `0`
- C. `undefined`
- D. `NaN`
> [!success]- Explanation
> The callback is a **regular function**, not an arrow. When `setTimeout` invokes it, there's no explicit receiver, so in non-strict mode `this` becomes the global object (`window`). The line `this.value = 99` ends up writing to `window.value`, not to the `Box` instance.
>
> So `obj.value` stays `0`.
>
> Fixes:
> 1. Arrow function: `setTimeout(() => { this.value = 99; }, 1000);` — arrow inherits `this` from `Box`'s call.
> 2. `bind`: `setTimeout(function() { this.value = 99; }.bind(this), 1000);`
> 3. Capture: `const self = this; setTimeout(function() { self.value = 99; }, 1000);`
>
> ---
>
> **→ Answer: B**

## Section L — ES2019 Modules and the no-dynamic-import trap (from Q31)
 
### Q28. In ES2019, which of the following is **not** part of the standardized module system?
- A. `import { x } from './mod.js'`
- B. `import './side-effect.js'`
- C. `import('./dynamic.js').then(...)`
- D. `export default value`
> [!success]- Explanation
> Dynamic `import()` is an **ES2020** feature, not ES2019. In ES2019:
> - Static `import` (specific bindings, default, namespace `*`, side-effect) — all standard.
> - `export default`, named `export`, re-export — all standard.
> - **Dynamic `import()`** — proposal still at Stage 3 in 2019; standardized in ES2020.
>
> If a question is strictly ES2019-scoped and asks about "dynamic module loading", the answer is "not available natively — use a bundler/loader extension or top-level static imports".
>
> ---
>
> **→ Answer: C**

### Q29. What is the key behavioral difference between CommonJS `require` and ES Modules `import` in **ES2019**?
- A. `require` is asynchronous; `import` is synchronous
- B. `require` returns a snapshot value; `import` provides live bindings to the source
- C. `require` works in browsers; `import` doesn't
- D. They behave identically; the difference is purely syntactic
> [!success]- Explanation
> The fundamental difference is **live bindings**. ES Modules export bindings, not values:
>
> ```javascript
> // counter.js
> export let n = 0;
> export const inc = () => { n++; };
>
> // main.js
> import { n, inc } from './counter.js';
> console.log(n); // 0
> inc();
> console.log(n); // 1 — the importer sees the update
> ```
>
> With CommonJS, `module.exports = { n, inc }` exports the value of `n` at that moment; mutations on the export side don't propagate to the importer's view of `n` (though if you exported an object reference, mutations on that object are visible — that's a different mechanism, not live binding).
>
> Other differences:
> - A: false. Both are typically synchronous in their respective environments. (CommonJS is fully synchronous; ESM has async parsing/instantiation but the user-visible execution is synchronous unless you use dynamic `import()`.)
> - C: false. Both can run in browsers (with bundlers for CommonJS, natively for ESM with `<script type="module">`).
>
> ---
>
> **→ Answer: B**

### Q30. What does the following code output?
```javascript
// counter.mjs
export let count = 0;
export function inc() { count++; }
 
// main.mjs
import { count, inc } from './counter.mjs';
inc();
inc();
console.log(count);
count = 999; // line X
```
- A. `2`, then line X succeeds and reassigns `count` to `999`
- B. `2`, then line X throws `TypeError` ("Assignment to constant variable" or similar)
- C. `0`, because importers see the original value
- D. `undefined`, because `count` is not exported as a binding
> [!success]- Explanation
> Two takeaways here:
>
> 1. **Live bindings (read side)**: `console.log(count)` correctly reflects the source module's mutations after `inc()` runs twice → prints `2`.
> 2. **Read-only on the importer side**: imported bindings cannot be reassigned by the importer. Trying `count = 999` throws `TypeError` (in strict mode, which is the default for ES Modules).
>
> So "live binding" means: importers always see the latest value, but **only the source module is allowed to mutate it**. The importer is a read-only window into the source's binding.
>

## Section M — `default` + named exports (from Q32)
 
### Q31. Which import statement is the **idiomatic** way to bring in the default export together with the named exports of the following module?
```javascript
// utils.js
export const PI = 3.14;
export function area(r) { return PI * r * r; }
export default class Shape {}
```
- A. `import { Shape, PI, area } from './utils.js';`
- B. `import Shape, PI, area from './utils.js';`
- C. `import Shape, { PI, area } from './utils.js';`
- D. `import { default as Shape, PI, area } from './utils.js';`
> [!success]- Explanation
> The idiomatic form is C: default import (no braces, name freely chosen) **before** the named imports (with braces, names must match).
>
> - A: invalid. `Shape` is the default export, not a named one. There's no `export { Shape }` in the source, so `import { Shape, ... }` fails to resolve.
> - B: syntax error. Named imports must be wrapped in braces.
> - C: the canonical pattern.
> - D: technically **valid** — `import { default as X, ... }` is a legitimate alternative spelling, since `default` is treated as a named export under the hood. But it's not idiomatic and almost never seen in real code. The question asks for the idiomatic form, so C wins.
>
> Reasoning: D works, but the question is about convention, not bare legality. If a question asked "which is invalid", D would not qualify as invalid.
>

> **→ Answer: C**

### Q32. Which of these is **invalid** syntax in ES2019?
- A. `export default 42;`
- B. `export default function () {};` (anonymous)
- C. `export default const x = 1;`
- D. `export { x as default };`
> [!success]- Explanation
> `export default` accepts an **expression** (or a function/class declaration), but **not** a `let`/`const`/`var` declaration. The reason: `default` exports the value/binding right there, and `const x = 1` is a declaration whose value would be the variable, not `1`.
>
> The fix is to split:
> ```javascript
> const x = 1;
> export default x;
> // or:
> export { x as default };
> ```
>
> - A: expression — fine.
> - B: anonymous function declaration is allowed in `export default` context.
> - D: re-exporting a named binding as the default — fine.

## Section N — Iterables and the iterator protocol (from Q35)
 
### Q33. Which of the following is **not** an iterable in ES2019?
- A. `'abc'`
- B. `new Map([['a', 1]])`
- C. `{ a: 1, b: 2 }`
- D. `arguments` (inside a function)
> [!success]- Explanation
> A plain object literal `{ a: 1, b: 2 }` does **not** implement `[Symbol.iterator]`, so it cannot be used with `for...of`, spread `[...obj]`, `Array.from(obj)` (without a mapper), or destructuring as iterable.
>
> The iterables in the language: `String`, `Array`, `Map`, `Set`, `TypedArray`, `arguments`, `NodeList` (DOM), and any object with a `[Symbol.iterator]` method.
>
> To iterate a plain object's entries, you go through `Object.keys(obj)`, `Object.values(obj)`, or `Object.entries(obj)` — all of which return arrays (which are iterable).
>
> ---
>
> **→ Answer: C**

### Q34. What does the following code output?
```javascript
const range = {
  from: 1,
  to: 3,
  [Symbol.iterator]() {
    let cur = this.from;
    const last = this.to;
    return {
      next() {
        return cur <= last
          ? { value: cur++, done: false }
          : { value: undefined, done: true };
      }
    };
  }
};
const result = [...range, ...range];
console.log(result);
```
- A. `[1, 2, 3]`
- B. `[1, 2, 3, 1, 2, 3]`
- C. `[1, 2, 3, undefined]`
- D. `TypeError: range is not iterable`
> [!success]- Explanation
> Each spread invokes `[Symbol.iterator]()` afresh, creating a new iterator with its own `cur` starting at `from`. So the second `...range` doesn't pick up where the first left off.

> Useful pattern: if you want a one-shot iterator (drains once and is done), have `[Symbol.iterator]` return `this` and put the state on the object itself. ## Section O — Generators, `yield`, `yield*` (from Q36)
 
### Q35. What does the following code output?
```javascript
function* gen() {
  console.log('a');
  yield 1;
  console.log('b');
  yield 2;
  console.log('c');
}
const it = gen();
console.log('before next');
it.next();
console.log('after next');
```
- A. `before next a after next` (generator runs eagerly when called)
- B. `a before next after next` (generator runs once when called)
- C. `before next a b c after next` (entire body runs on first `next`)
- D. `before next a after next` (only first `console.log` runs through to first yield)
> [!success]- Explanation
> Calling `gen()` does **not** execute the body — it returns a paused generator object. The body only runs when `.next()` is called, and execution proceeds until the next `yield` (which suspends), `return`, or end of function.
>
> Trace:
> 1. `console.log('before next')` → prints `before next`.
> 2. `it.next()` runs the body until `yield 1`: prints `a`, then suspends.
> 3. `console.log('after next')` → prints `after next`.
>
> If we then called `it.next()` again, it would print `b`, suspend at `yield 2`. Each `next()` advances exactly one segment.
>
> ---
>
> **→ Answer: D**

### Q36. What does the following code output?
```javascript
function* inner() {
  yield 'x';
  yield 'y';
  return 'inner-done';
}
function* outer() {
  yield 1;
  const r = yield* inner();
  yield r;
  yield 2;
}
console.log([...outer()]);
```
- A. `[1, 'x', 'y', 2]`
- B. `[1, 'x', 'y', 'inner-done', 2]`
- C. `[1, 'x', 'y', 'inner-done', 'inner-done', 2]`
- D. `[1, ['x', 'y'], 'inner-done', 2]`
> [!success]- Explanation
> Two key facts about `yield*`:
>
> 1. It **delegates** iteration to the inner iterable, yielding each of its values to the outer consumer (so `'x'` and `'y'` flow through to the spread).
> 2. The inner generator's `return` value is **the value of the `yield*` expression** — not yielded, but assigned. So `r === 'inner-done'`.
>
> Then `yield r` produces `'inner-done'`, then `yield 2` produces `2`.
>
> Final spread: `[1, 'x', 'y', 'inner-done', 2]`.
>
> Note that `'inner-done'` appears once in the output — from `yield r`, not from `yield*` itself. The `done: true` value of an inner generator is captured but not auto-yielded.
>
> ---
>
> **→ Answer: B**

### Q37. What does `it.next('hello')` do in the following code?
```javascript
function* gen() {
  const received = yield 'first';
  yield received;
}
const it = gen();
console.log(it.next().value);   // (1)
console.log(it.next('hello').value); // (2)
```
- A. `(1) 'first', (2) 'hello'`
- B. `(1) 'first', (2) 'first'`
- C. `(1) undefined, (2) 'hello'`
- D. Throws — `next` only takes zero arguments
> [!success]- Explanation
> The argument to `next()` becomes the **value of the previous `yield` expression** when the generator resumes:
>
> 1. First `it.next()` (no argument): runs until `yield 'first'`, returns `{ value: 'first', done: false }`. `received` is not yet assigned.
> 2. `it.next('hello')`: resumes; the `yield 'first'` expression evaluates to `'hello'`, so `received = 'hello'`. Continues to `yield received`, returns `{ value: 'hello', done: false }`.
>
> The argument to the *first* `next()` call is always discarded — there's no preceding `yield` to receive it. This bidirectional communication is what enables coroutine-style flows with generators.
>
> ---
>
> **→ Answer: A**

## Section P — `async/await` and rejected promises (from Q41)
 
### Q38. What does the following code output?
```javascript
async function fail() {
  throw new Error('boom');
}
async function caller() {
  try {
    await fail();
    console.log('after await');
  } catch (e) {
    console.log('caught:', e.message);
  }
}
caller();
```
- A. `caught: boom`
- B. `after await`
- C. `caught: boom`, then `after await`
- D. Unhandled promise rejection
> [!success]- Explanation
> An `async` function that throws returns a **rejected promise**. Awaiting that rejected promise re-throws synchronously (from the awaiter's point of view), so `try/catch` catches it. The line `console.log('after await')` is unreachable because control jumps to the catch block.
>
> Mental model: `await p` is roughly equivalent to `if (p is fulfilled) value = p.value else throw p.reason`. So `try/await/catch` works exactly like `try/sync-call/catch`.
>
> ---
>
> **→ Answer: A**
 
---
 
### Q39. What does the following code output?
```javascript
async function f() {
  try {
    return Promise.reject(new Error('inside'));
  } catch (e) {
    return 'caught';
  }
}
f().then(
  v => console.log('fulfilled:', v),
  e => console.log('rejected:', e.message),
);
```
- A. `fulfilled: caught`
- B. `rejected: inside`
- C. `fulfilled: [object Promise]`
- D. Unhandled promise rejection
> [!success]- Explanation
> Trap question. **`return Promise.reject(...)` is not the same as `throw`** — and crucially, `try/catch` does **not** catch a returned rejected promise.
>
> Why: `try/catch` catches synchronous throws. A `return` statement is not a throw — the function exits normally with the rejected promise as its return value. The `async` function then unwraps that rejected promise as its own rejection.
>
> To make `try/catch` work, you need to `await` the rejection (turning it into a synchronous throw):
> ```javascript
> try {
>   return await Promise.reject(new Error('inside'));
>   //     ^^^^^ this is the fix — known as "return await"
> } catch (e) { ... }
> ```
>
> This is one of the few places where `return await` is meaningful (most of the time it's redundant).
>
> ---
>
> **→ Answer: B**

### Q40. What does the following code output?
```javascript
async function load() {
  const results = await Promise.all([
    Promise.resolve(1),
    Promise.reject(new Error('fail')),
    Promise.resolve(3),
  ]);
  return results;
}
 
load().then(
  v => console.log('got:', v),
  e => console.log('err:', e.message),
);
```
- A. `got: [1, undefined, 3]`
- B. `got: [1, Error: fail, 3]`
- C. `err: fail`
- D. Unhandled rejection
> [!success]- Explanation
> `Promise.all` is **fail-fast**: as soon as any input rejects, the resulting promise rejects with that reason, and all other results are discarded. So `await Promise.all(...)` throws inside `load`, which propagates as a rejection of `load()`.
>
> The error handler in `.then(onF, onR)` receives the error.
>
> Compare:
> - `Promise.all`: rejects on first failure (ES2015). ✓ in scope.
> - `Promise.race`: settles with the first to settle (success or failure).
> - `Promise.allSettled` (ES2020): never fails — returns `{ status, value/reason }` for each.
> - `Promise.any` (ES2021): fulfills on first success; only rejects if all fail.
>
> Within ES2019, `Promise.all` and `Promise.race` are your only built-in combinators. Anything more nuanced you compose manually.
>
> ---

## Quick Reference Cheat Sheet
 
For at-a-glance review (kept here for the answer-side, since this whole document is the answer side):
 
**`Object.defineProperty` defaults**: `writable: false`, `enumerable: false`, `configurable: false`. Direct assignment defaults: all `true`.
 
**Coercion `Number()` results**: `null → 0`, `undefined → NaN`, `'' → 0`, `[] → 0`, `{} → NaN`.
 
**Equality algorithms**:
- Strict (`===`): `+0 === -0` true, `NaN === NaN` false. Used by `switch`, `indexOf`.
- SameValueZero: `+0` and `-0` equal, `NaN` equal to `NaN`. Used by `Map`/`Set` keys, `Array.prototype.includes`.
- SameValue (`Object.is`): distinguishes `+0` and `-0`, `NaN` equal to `NaN`. Used by descriptor comparisons.
**Immutability ladder**: `preventExtensions` ⊂ `seal` ⊂ `freeze`.
 
**Arrow functions lack**: own `this`, own `arguments`, own `super`, own `new.target`, `prototype`, `[[Construct]]`.
 
**`WeakMap` / `WeakSet` API**: only `set/get/has/delete` (Map) or `add/has/delete` (Set). No iteration, no `size`, no `clear`. Keys must be objects (in ES2019).
 
**ES2019 vs ES2020+ boundary**:
- ✓ ES2019: `Array.flat`/`flatMap`, `Object.fromEntries`, `String.trimStart`/`trimEnd`, `Symbol.description`, optional catch binding, stable `Array.sort`, `Function.toString` revision.
- ✗ ES2020+: dynamic `import()`, `??`, `?.`, `Promise.allSettled`, `BigInt` literal `n` (BigInt itself is ES2020), `globalThis`, `String.matchAll`, `String.replaceAll` (2021), private class fields (2022).
**Generator delegation**: `yield*` forwards every value of an inner iterable; the inner generator's `return` value becomes the value of the `yield*` expression.
 
**`await` semantics**: `await rejected` throws; `return rejected` does not (need `return await rejected` to make `try/catch` see it).
